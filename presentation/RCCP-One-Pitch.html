Proposed & led by Dominik Sady · Capacity Planning Gravesend · UKP1 For management review & discussion
01 Where we are today

Capacity planning lives in a spreadsheet

It works — until it doesn’t. The current process is manual, fragile, and impossible to audit. As volume and complexity grow, the risk grows with it.

Trust

No validation

Bad or mistyped data flows straight into the plan. Nothing checks it before decisions are made.

Versions

Which file is current?

Copies multiply across inboxes and drives. No single, agreed “published” plan.

Effort

Slow & manual

Hours of rebuilding every cycle. Time spent maintaining the sheet, not analysing the plan.

Insight

No real scenarios

“What if we add overtime?” means another tab and more formulas — not a quick, reliable answer.

Audit

No history

No record of who changed what, or what last month’s plan actually said.

Continuity

Key-person risk

The logic lives in one person’s head and one fragile file. If they’re away, planning stalls.

A3 Technical · Security & access

Internal, authenticated, auditable

Access control

  • Username + password login, issuing a signed token (JWT / HS256).
  • Two roles: admin (full) and user (planning workflow).
  • Passwords hashed (bcrypt) — never stored in plain text.

Data & network

  • Runs on the company network — not exposed to the public internet.
  • Planning data never leaves UKP1 infrastructure.
  • Database access via a dedicated SQL login; secrets held in environment config / the deployment runbook, not in code.

Auditability: every upload, validation result and published baseline is recorded with the user and timestamp — a complete history of who changed what, and when.

Low footprint, easy to support

Environment

HostWindows VM on the UKP1 network RuntimePython (FastAPI via uvicorn) + a static React build DatabaseSQL Server (existing company instance) DependenciesODBC Driver 17 for SQL Server; Python & Node toolchains

  • Backups: standard SQL Server backups + the uploads folder.
  • Restore / move: redeploy from source + restore the DB — documented step-by-step.
  • Footprint: single small VM; no external services or licences.
  • Source control: code in Git; reproducible builds.

What IT would need to provide: a Windows VM (or container host) on the network, a SQL Server database & login, and a backup schedule. A deployment runbook already exists.

A5 Technical · Data model (high level)

How the data is organised

Reference

Masterdata

Lines, plants, items/SKUs, pack types, labour pools, fill-speeds, line & plant resource requirements, warehouse capacity. Changes slowly; managed separately.

Per cycle

Planning data

Production orders, demand plan, master stock, capacity calendar, headcount plan, portfolio changes, actuals. Scoped to each batch and versioned.

Process

Workflow

Import batches, uploaded files, validation results, published baselines (plan versions). The audit trail.

Computed, never stored: derived figures — net capacity hours, litres-per-minute, allocated stock — are calculated in database views so the numbers are always consistent and can’t drift.
